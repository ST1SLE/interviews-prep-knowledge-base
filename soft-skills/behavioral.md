## Q1. "Расскажи про свой самый сложный проект" (STAR Framework)

> "**STAR** — структура ответа на behavioral вопросы:
>
> - **S**ituation — контекст (1 предложение)
> - **T**ask — задача, которую нужно было решить
> - **A**ction — конкретные действия (стек, решения, trade-offs)
> - **R**esult — измеримый результат
>
> Ключевое: говори о **своих** действиях ('я сделал'), не о команде ('мы сделали'). Интервьюер хочет понять твой личный вклад."

**Заготовки для Михаила:**

| Проект | S (Situation) | T (Task) | A (Action) | R (Result) |
|--------|--------------|----------|------------|------------|
| **Text2Brainrot** | Учебный проект на ФПМИ, нужна платформа конвертации текста в видео | Спроектировать full-stack систему с NLP + TTS + video generation | FastAPI + Celery + Redis для async задач, GigaChat API для генерации, Edge-TTS для озвучки, PostgreSQL для хранения | Оценка **10/10**, платформа работает end-to-end |
| **Альфа-Банк** | Стажировка, нужна модель кредитного скоринга | Построить предсказание дефолта по заявкам | Feature engineering (агрегаты по транзакциям), CatBoost, подбор гиперпараметров через Optuna | Улучшение Gini модели, production-ready pipeline |
| **Random Coffee** | Бот для МФТИ, нужен умный мэтчинг участников | Реализовать семантическое сопоставление по интересам | Sentence-BERT для эмбеддингов профилей, косинусное сходство, Telegram Bot API | Бот работает: @random_coffee_mipt_bot |

## Q2. "Объясни градиентный бустинг продакт-менеджеру"

> "Техническая эмпатия — умение объяснить сложное простыми словами. Правило: **ноль жаргона**.
>
> **Аналогия для градиентного бустинга:**
> 'Представь, что ты угадываешь цену квартиры. Первый эксперт даёт грубую оценку. Второй эксперт смотрит, где первый ошибся, и корректирует. Третий корректирует ошибки второго. И так далее — каждый следующий эксперт специализируется на ошибках предыдущих. В итоге команда из 100 слабых экспертов даёт очень точную оценку.'
>
> **Аналогия для нейросети:**
> 'Как ребёнок учится отличать кошку от собаки: сначала ошибается, потом получает обратную связь, корректирует — и со временем точность растёт. Нейросеть так же: смотрит на тысячи примеров, ошибается, корректирует свои внутренние настройки.'
>
> Практика: подготовить 30-секундное объяснение для каждого ключевого алгоритма."

| Алгоритм | Аналогия для PM |
|----------|----------------|
| Градиентный бустинг | Команда экспертов, каждый исправляет ошибки предыдущего |
| Нейросеть | Ребёнок учится на примерах, постепенно улучшаясь |
| Кластеризация | Разложить вещи по кучкам по похожести |
| A/B тест | Показать половине клиентов версию A, половине B, сравнить результат |
| Embeddings | 'Координаты смысла' — похожие слова рядом в пространстве |

## Q3. "Сложная нейросеть vs простая интерпретируемая модель — что выбрать?"

> "Trade-off: точность vs интерпретируемость. Ответ зависит от контекста:
>
> **Когда интерпретируемость важнее:**
> - Финансы: регуляторные требования (ЦБ требует объяснить отказ в кредите)
> - Медицина: врач должен понять, почему модель рекомендует диагноз
> - Высокие ставки: решение влияет на жизнь человека
>
> **Когда точность важнее:**
> - Рекомендации: пользователю неважно ПОЧЕМУ этот фильм, важно чтобы понравился
> - Computer Vision: распознавание изображений, где интерпретация не нужна пользователю
>
> **Компромисс**: сложная модель + explainability tools (SHAP, LIME). В Альфа-Банке: CatBoost для скоринга + SHAP values для объяснения каждого решения → удовлетворяет и точность, и регуляторные требования.
>
> Правильный ответ на собесе: 'Зависит от контекста. Какие требования к интерпретируемости? Какова цена ошибки? +0.01 Gini может не стоить потери прозрачности.'"

## Q4. "Что делал с грязными данными / нехваткой разметки?"

> "Инженерный прагматизм — начать с simple baseline, итеративно улучшать.
>
> **Грязные данные:**
> 1. EDA: понять масштаб проблемы (% пропусков, выбросы, дубликаты)
> 2. Не тратить месяцы на чистку — начать с simple model на том, что есть
> 3. Итеративно: модель показывает, какие фичи важны → чистим именно их
>
> **Нехватка разметки:**
> 1. Rule-based baseline: простые правила дают 60-70% accuracy → уже полезно
> 2. Semi-supervised: обучаем на малой размеченной выборке, предсказываем на неразмеченной, добавляем уверенные предсказания в train
> 3. Active learning: модель сама выбирает, какие примеры разметить (наиболее неопределённые)
> 4. Data augmentation: расширяем train-набор трансформациями
>
> Связь с реальным опытом: в Альфа-Банке данные были неидеальными — пропуски в кредитных историях, дубликаты заявок. Пропуски сами по себе были информативным признаком для CatBoost."

## Q5. "Конфликт с коллегой по выбору архитектуры — как решали?"

> "От спора к эмпирической проверке. Шаблон ответа:
>
> 1. **Выслушал** аргументы коллеги — понял его точку зрения
> 2. **Предложил** объективный критерий: 'Давай проверим на данных'
> 3. **Провёл** эксперимент: кросс-валидация / A/B тест / прототип
> 4. **Решение** на основе данных, не эмоций
>
> Пример для Михаила: 'В проекте Random Coffee стоял выбор между TF-IDF и Sentence-BERT для мэтчинга. Вместо спора написал скрипт, который оценил качество мэтчей по обоим подходам на тестовой выборке. Sentence-BERT показал значительно лучшие результаты по косинусному сходству на парах с известной совместимостью → выбрали его.'
>
> Ключевое для интервьюера: ты решаешь конфликты через данные, а не через авторитет или эмоции."
